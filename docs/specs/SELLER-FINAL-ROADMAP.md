# SELLER-FINAL — 추후 확장 명세서 (post-MVP)

> **버전**: v1.0.0 · **기준일**: 2026-07-20 · **상태**: 백로그
> **결정(2026-07-20 사용자)**: 시맨틱 캐시·RAG(구 4-4)는 MVP 범위에서 제외하고 post-MVP
> 확장으로 이동한다. MVP 는 4-3(분석 이력)까지로 마감 — 기능 동작에 결손 없음(캐시·RAG 는
> 품질·비용 최적화 계층). 본 문서가 확장 백로그의 정본이다.

## E1. 시맨틱 캐시 (question_cache) — 구 4-4

- **목적**: 유사 질문 + 동일 기간 + 당일이면 LLM 미실행으로 이전 보고서 반환 — 비용·지연 절감 + 같은 질문에 다른 답 방지(일관성 장치 ⑧).
- **설계 요약**: pg-catalog(5433) pgvector 컬렉션 `question_cache` — 질문 임베딩이 조회 키, metadata 에 sellerId·기간·참조키. 히트 판정(유사도 임계·동일 기간·당일)은 코드. 미스면 이전 보고서를 앵커로 주입.
- **분석 이력과의 관계**: 이력(pg-profile)은 기록(보존·정확 조회), 캐시(pg-catalog)는 절약 장치(만료 가능) — 저장물은 겹치지만 키·수명이 다르다. HistoryEntry 의 기간 필드가 "동일 기간" 판정 재료로 재활용 가능.
- **선행 조건**: ANTHROPIC_API_KEY + 셀프호스트 한국어 임베딩(1024-dim, `uv sync --group embedding`) 로컬 구동 확인 + pg-catalog pgvector.
- **수용 기준**: 동일 질문 2회차 LLM 호출 0 · 기간 다르면 미스 · 판매자 격리.

## E2. RAG 지식 베이스 (analysis_guides 외) — 구 4-4

- **목적**: 분석 기준서(용어·산식 정의)를 워커에 고정 주입 — "분석 전 기준서 검색, 코드 판정 번복 금지"(장치 ③).
- **설계 요약**: pg-catalog pgvector 컬렉션 `analysis_guides`·`policies`·`product_docs`. 인제스트는 `app/pipelines/seller_kb.py`(오프라인) — 쓰기 권한은 인제스트 경로만, 에이전트에는 `search_analysis_guide` @tool 만 노출(현재 degrade 스텁 — 인터페이스는 이미 배선됨).
- **차단 요인 🔴**: **분석 기준서 원천 문서가 아직 없다**(팀 산출물 선행). 문서 없이는 착수 불가.
- **수용 기준**: 스텁 문구 소멸 · 워커 보고서에 기준서 근거 인용 · 인제스트 재실행 멱등.

## E3. 사용자 행동 통계 기반 로그데이터 생성 (신규 — 2026-07-20 사용자 추가)

- **목적**: 실 트래픽이 쌓이기 전, 데모·스모크·분석 품질 튜닝에 쓸 **현실적인 행동 로그 시드**를 만든다 — 분석 워커가 "빈 데이터 degrade"가 아니라 실제 패턴(요일 효과·가격 인상 후 이탈·품절 급감 등)을 보고 말하게.
- **설계 방향**:
  - 대상: behavior_events 8종(F4)·order_status_logs·product_change_logs(F5 기록 규칙 준수 — 구매확정·ORDERED 미기록, 품절 신호=STOCK→0 등).
  - 생성기: 시나리오 파라미터(기간·상품 수·전환율 곡선·이상 이벤트 주입)를 받아 Spring DB 시드 스크립트 또는 I-* mock 서버 응답을 산출. 실사용자 통계(공개 커머스 벤치마크·팀 서비스 실측)로 분포 보정.
  - 배치 위치는 BE 와 협의 필요(Spring DB 소유는 BE) — AI 저장소 원칙(원본 사본 금지) 위반하지 않도록 **생성기는 도구, 데이터는 Spring 측 주입**.
- **수용 기준**: 시드 주입 후 스모크 분석 발화가 degrade 아닌 실수치 보고서 산출 · 이상 시나리오(매출 급락 주입)를 sales_anomaly 가 탐지.

## E4. 일관성 회귀 (같은 질문 10회)

같은 질문 10회 실행 → 결론 일치율 측정(t=0·구조화 출력·코드 판정이 실제로 일관성을 주는지 실증). 캐시(E1) 도입 전 기준선 확보용으로 먼저 실행할 가치 있음.

## E5. 차트 전달 (chart_agent 복원)

SSE 4종에 차트 경로 없음 — 새 이벤트 or 파일 URL 계약 **정본 개정 선행**(계약 우선 규칙). `AnalysisFinding.chart_data_hint` 는 복원 대비 보존돼 있음.

## E6. BE 협력 요청 (확장 시점에 같이)

- **I-9 단건 조회**(GET .../products/{productId}) — stale 검증·추천 적용의 페이지 순회(RISKS T1) 제거.
- behavior_events 원시 접근이 아닌 **집계 API 확장**(현행 원칙 유지) — E3 시드와 짝.

## E7. 운영 정리 작업

- **draft checkpoint 청소 배치**: 만료(TTL 경과)·완료 스레드 주기 삭제(RISKS T4).
- **분석 이력 per-item 재검토**: 동시 쓰기 경합(T3)·`analysis_detections` 워커별 상세 테이블(SPEC §9.1 잔여) — 이력 조회 UI 가 생기면 페이지네이션과 함께.
- **mask_output 스트림 버퍼링**: 청크 경계 패턴 누락(T2) — 꼬리 버퍼 방식.
- **두 기간 비교 분석**(R3 승계): "지난주 대비 이번주" — 워커 입력 계약 확장 필요.

## 우선순위 제안

| 순위 | 항목 | 이유 |
|---|---|---|
| 1 | E3 로그데이터 생성 | 이것 없이는 E1/E2/E4 의 효과 검증도, 데모도 공허하다 — 모든 확장의 토대 |
| 2 | E4 일관성 회귀 | 캐시 도입 전 기준선. API 키만 있으면 저비용 |
| 3 | E1 시맨틱 캐시 | 비용·일관성 즉효. 이력 스키마 재활용으로 착수 비용 낮음 |
| 4 | E2 RAG | 팀 기준서 문서 산출 대기 — 문서가 나오는 시점에 착수 |
| 5 | E5~E7 | 운영 안정화·BE 협의 종속 |
