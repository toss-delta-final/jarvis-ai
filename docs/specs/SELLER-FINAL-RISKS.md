## 1. BE 확정 대기 (🔴 — 가정으로 동작 중)

| # | 항목 | 현재 가정 | 확정 시 수정 지점 |
|---|---|---|---|
| B1 | **신원 헤더명** `X-Seller-Id`/`X-Brand-Id` | AI측 제안값으로 구현 | `app/api/deps.py` `require_seller_internal` 한 곳 |

## 3. 기술 리스크 (알고 수용한 것 — 각 항목에 완화책 존재)

| # | 리스크 | 완화/수용 근거 |
|---|---|---|
| T1 | **I-9 productId 필터 부재** — stale 검증·추천 적용이 페이지 순회(기본 20×10=200건). 상품 200+ 판매자는 미발견→stale 오탐 | Settings 로 상한 조정 가능. 근본 해결은 BE 에 단건 조회(I-9b) 요청 — ROADMAP E6 |
| T2 | **mask_output 청크 경계** — 시크릿 패턴이 스트림 청크에 걸치면 누락 가능 | 보고서·결과는 단일 문자열 적용이라 무풍. general 스트림만 해당 — 버퍼링 개선 보류 승계 |
| T3 | **이력 동시 쓰기 경합** — 같은 판매자 동시 분석 2건이면 마지막 쓰기 승리 | 이력은 부가 데이터·발생 확률 낮음. post-MVP 에서 per-item 키 재검토 |
| T4 | **draft checkpoint 무청소** — 만료·미승인 draft 스레드가 pg-profile 에 누적 | TTL 로 실행은 차단됨(안전 무관). 청소 배치는 ROADMAP E7 |
| T5 | **dev 폴백의 운영 유출** — InMemory saver/store 는 재시작 시 draft·이력 증발 | auth_mode=jwks 에서 폴백 금지(기동 실패로 드러남). 운영 배포 체크리스트에 auth_mode 확인 필수 |
| T6 | **planner 입력 증가** — 이력 5건 주입으로 토큰·오분류 영향 가능성 | 블록 라벨([이번 질문])로 분리. 스모크에서 이력 유/무 발화 비교 확인 항목에 추가 권장 |
| T7 | **90s 초과 가능성** — 분석 5종 + 검증 루프 3회 최악 경로 | 워커 병렬 + 개별 60s 상한. 실측 후 §2.9 개정 제안 여부 판단(SPEC §12 승계) |
| T8 | 적용 발화 정규식의 변형 미커버("첫 번째 적용해줘" 등) | 미일치는 supervisor→product→되묻기로 수렴(오실행 0). 커버리지 확장은 스모크 데이터 보고 판단 |

## 5. 우선순위 요약 (다음 세션 첫 확인)

1. API 키 발급 → 스모크(V1·V2) — 이것 없이는 "잘 작동" 판정 불가.
2. 로컬 분리 커밋(D4) — 작업량이 워킹트리에만 있는 상태가 최대 유실 위험.
3. BE 회신 B1·B2 — 각 한 곳 수정으로 흡수 가능하게 설계돼 있음.
